<a id="三大模型核心知识"></a>
## 三、大模型核心知识图谱

### 3.1 Transformer 架构（面试必问）

```
输入 Token → Token Embedding + Position Embedding → N×Transformer Block
                                                       ↓
                                              Self-Attention（QKV）
                                                       ↓
                                              Feed Forward Network
                                                       ↓
                                          LayerNorm + Residual Connection
                                                       ↓
                                        → 输出 Logits → Softmax → Token 预测
```

**核心要点（面试必问）：**
- Self-Attention 的时间复杂度是 O(n²·d)，n 是序列长度，d 是隐层维度
- Flash Attention：将 O(n²) 显存降到 O(n)，通过分块计算+算子融合实现
- 位置编码：RoPE（旋转位置编码，Llama/Qwen 使用）比绝对位置编码更适合长文本

**面试口述版：**
> "Attention 的核心是让每个 token 和序列中所有 token 计算相关性分数。标准实现需要 O(n²) 的显存存储注意力矩阵，Flash Attention 的思想是把序列分成块，在块内计算后逐步累加到全局，避免一次性申请巨大的显存矩阵，这是训练 7B 以上模型的必备优化。"

### 3.2 LLM 训练三阶段

| 阶段 | 数据 | 目标 | 典型方法 |
|------|------|------|---------|
| Pretraining | 万亿级网页/书籍语料 | 学习语言能力和世界知识 | Next Token Prediction |
| SFT（监督微调）| 万级高质量指令数据 | 注入任务能力、格式化输出 | 全量微调 / LoRA |
| RLHF | 人类偏好数据 | 对齐人类偏好、提升无害性 | PPO / DPO / ORPO |

**DPO（Direct Preference Optimization）——2024年以来的主流选择：**
> "DPO 的核心思想是：不训练 reward model，直接用偏好数据优化策略。相比 PPO，DPO 简化了流程、降低了训练不稳定性、显存占用减少约 40%。主流开源模型（Llama3、Qwen2）的对齐基本都用了 DPO。"

## 四、大模型微调专题（你的核心技术）

### 4.1 微调方法全景图

```
全量微调（Full Fine-tuning）
    ↓ 显存问题
Adapter系列：LoRA / QLoRA / AdaLoRA / DoRA
    ↓ 组合使用
LoRA + QLoRA（全量+量化，最实用）← 你的项目用这个
```

### 4.2 LoRA 核心参数（结合你的项目讲）

```python
from peft import LoraConfig, get_peft_model

config = LoraConfig(
    r=16,                      # 秩（Qwen2.5项目用 r=16）
    lora_alpha=32,             # 缩放因子 = 2×r
    target_modules=[            # Qwen2.5 的 QKV + FFN 全注入
        "q_proj", "k_proj", "v_proj", "o_proj",
        "gate_proj", "up_proj", "down_proj"
    ],
    lora_dropout=0.05,
    bias="none",
    task_type="CAUSAL_LM",
)
```

**面试口述版（结合你的两个项目）：**
> "两个项目我都用过 LoRA。智慧港口项目用的是标准 LoRA（Qwen-14B，r=16），因为那时显存相对宽裕。物流枢纽项目更复杂：基座是 Qwen2.5-14B，且要在 RTX 5090 上跑，必须做量化。我用 QLoRA——即在 LoRA 基础上，把原模型量化到 INT8/FP16，只对 LoRA 权重保持高精度训练。NF4（4-bit NormalFloat）量化能把 14B 模型从 28GB 压缩到约 16GB，单卡可训。"

**为什么 QLoRA 在 RTX 5090 上可行？**
> "RTX 5090 是新架构，显存带宽和算力都大幅提升。QLoRA 的 NF4 量化在保持模型效果的同时，把显存占用减半。这让单卡微调 14B 模型成为工程上可行的选择。"

### 4.3 你的 8-bit 量化经验（面试加分项）

> "8-bit 量化不是简单压缩，还有两个关键工作：① 异常值处理——LLM 的权重分布通常是'少数值很大，大多数值很小'的，量化时需要对异常值单独处理，否则精度损失大；② 显存优化——不只是压缩模型权重，还有 KV Cache 的量化。我用 bitsandbytes 库的 LLM.int8() 实现，配合 QLoRA 框架使用。"

### 5.1 RAG 完整流程（结合你的两个项目）

```
用户问题
   ↓
Query 改写 / 扩展（可选）← 物流枢纽项目用到了
   ↓
检索（Retrieval）
  ├─ 向量检索（BGE → FAISS）← 两个项目都用

   ↓
检索结果 → 重排序（CrossEncoder）← 两个项目都用
   ↓
Context 组装（分数增强）← 港口项目特色
   ↓
LLM 生成（GPT-4 / Qwen2.5）
```

```
财务用户查询（如："Q3各区域毛利率对比"）
    ↓
意图理解（Intent Classification）← 你做的 Prompt 优化
    ↓
Schema 匹配（哪个表？哪些字段？）← 你做的 Schema 注入
    ↓
CoT 推理链路（4-6步）← 你设计的推理框架
    ↓
Tool Calling → Snowflake SQL 查询
    ↓
结果验证（语法校验 + 业务逻辑校验）
    ↓
自然语言回答 + 数据可视化
```

### 6.2 CoT vs ReAct（你的理解）

| | CoT（Chain of Thought）| ReAct（Reasoning + Acting）|
|---|---|---|
| 特点 | 纯推理，不调用外部工具 | 推理 + 工具调用交替 |
| 适用 | 数学、逻辑分析 | 需要查数据的任务 |
| 你的应用 | 财务指标计算 | RAG 问答、Agent 任务执行 |

> "在江森自控，我们选择 CoT 而不是 ReAct，是因为财务分析的数据源是固定的（Snowflake），不需要搜索网页或查文件，CoT 的确定性更高。ReAct 更适合开放世界的任务——每次行动的结果是不确定的，需要根据反馈调整。"

### 6.3 Tool-use 接口设计（你的核心贡献）

> "Tool-use 的核心是接口设计。我定义了三个关键部分：① Tool Description——每个工具（如 execute_sql）的描述要告诉 LLM 什么场景用它；② Input Schema——每个工具的参数要有类型约束和示例，LLM 生成参数时更准确；③ Error Handling——工具执行失败时的处理策略，比如 SQL 执行失败后，Agent 应该尝试简化查询还是返回错误。"

## 八、2025-2026 前沿进展

### 8.1 核心技术趋势

**① OpenAI o1/o3 系列（推理时代）**
> "o1 的核心突破是：推理能力通过强化学习在训练阶段内化，而不是 prompt 技巧。o1 在国际数学奥林匹克达到银牌水平，o3 在 ARC-AGI 上接近人类水平。核心思想是 Test-time Compute Scaling——给模型更多'思考'时间（Thinking Token），算力越大推理越强。"

**② DeepSeek 系列（国产开源+低成本）**
> "DeepSeek-V2 的 MLA（Multi-head Latent Attention）大幅降低推理成本和 KV Cache 显存占用。DeepSeek-V3 的 MoE 架构（总参数 236B，激活 21B），训练成本只有同等规模模型的 1/10。国产开源模型的崛起让私有化部署成为主流选择。"

**③ Agentic AI（多 Agent 协作）**
> "2026 年最重要的方向是 Multi-Agent：多个专业 Agent 协作完成任务（如你项目 3 的 Supervisor 框架）。核心挑战是 Agent 之间的通信协议、任务分解策略、冲突解决机制。"

**④ 长上下文 + 多模态**
> "上下文窗口持续扩展（Gemini 1.5 支持 1M tokens）。多模态从'图像+文本'向'视频+3D+音频'扩展。原生多模态（Gemini/ChatGPT-4o）成为主流，替代'图像 Encoder + LLM'的拼接方案。"

<a id="九高频问题"></a>
## 九、面试高频问题及答法

### Q1：RAG 解决幻觉，怎么做到的？
> "RAG 本身只能保证'被检索到的知识'是准确的，但还有两类幻觉：① 检索到了但 LLM 推理错误；② 检索结果本身有噪音。我的方案是：多阶段验证（CrossEncoder 过滤 + 置信度阈值）+ 引用溯源（每个回答标注来源）+ 在江森自控学的 CoT 链路中嵌入结果校验步。"

### Q2：LoRA 和全量微调怎么选？
> "取决于显存和任务差异度。Qwen-14B 全量微调需要 8×A100，成本高。LoRA 只训 0.1% 参数，单卡可跑。我的两个项目都选了 LoRA：任务和预训练比较接近（都是中文问答），LoRA 足够；如果要注入全新知识（如从通用→法律），可能需要更多参数。"

### Q3：你在江森自控具体做了什么？
> "我做了三件事：① 设计了财务 Agent 的 CoT 推理链路，把分析师的复杂逻辑拆成 4-6 个标准步骤；② 优化 Prompt 和 Tool-use Schema，提升 Agent 生成 SQL 的执行成功率；③ 维护 Data Pipeline（Snowflake→ADF→Power BI），保障跨境财务数据的准确交付（100% 通过 SIT）。"

### Q4：模型推理速度慢怎么优化？
> "结合我的两个项目经验：① 量化——8-bit 量化（bitsandbytes），吞吐提升 2-4x；② 显存优化——RTX 5090 的 QLoRA 解决了 OOM；③ 批处理——FastAPI 后端做了请求合并；④ 缓存——热门问题走 Redis 缓存，避免重复 LLM 调用。"

### Q5：你的研究课题是什么？
> "我的研究方向是'基于 Transformer 与多源异构数据融合的碳价时序预测'。核心是把市场数据（价格/交易量）和文本数据（政策新闻）融合，用 Transformer 建模跨模态依赖关系。这让我对 Transformer 的时序建模能力和多模态融合有深入理解。"

### Q6：港口和物流两个项目的区别？
> "港口项目偏教育：核心创新是'分数增强算法'——结合学生画像做个性化响应，更像一个智能助教。物流枢纽项目偏业务：核心是端到端自动化——从 PDF 政策文件到量化评分，强调准确率和可审计性。两个项目的技术栈相似，但优化目标和评估标准完全不同。"
